# ML Serving Rules — Context Reference

> Backend Architect: read this before touching any ML serving code.

---

## Rule 1: .predict() Only

**Law:** All regression models use `.predict()`. Never `.predict_proba()`.

**Why:** HewaSense models are regressors (predicting continuous values like precipitation amounts). `.predict_proba()` is a classifier method. Calling it on a regressor raises `AttributeError`. Previously, this error was silently caught and fell through to a synthetic `_baseline_prediction` that returned fabricated values. That fallback has been removed (GOTCHA Law #1) — now a missing or broken model causes an explicit failure.

**Check:** `forecast_service.py` — `load_model()` reads from `active_model.json`. If the config or the model file is missing, it fails explicitly. No fallbacks.

---

## Rule 2: Feature Alignment (83 Features)

**Law:** Serving must generate the exact same 83 features that training selected.

**Why:** The training pipeline engineers ~245 features (after removing 121 leaky rainfall-derived features from ~279 candidates), then selects 83 via hybrid feature importance + source diversity. The serving path (`prepare_features`) must reproduce this exact pipeline via `feature_engineering.build_feature_vector()`. If serving generates fewer features, the model rejects the input due to shape mismatch.

**Canonical source:** `feature_selection_results.json`

**Check:** Both `train_pipeline.py` and `forecast_service.py` must reference `feature_selection_results.json`.

### Feature Categories (from training pipeline)
1. **Raw features** — direct observations (temp, precip, NDVI)
2. **Lag features** — values from N days ago (7, 14, 30, 60, 90-day lags)
3. **Rolling window stats** — mean, std, min, max over windows (7, 14, 30 days)
4. **Interaction features** — products/ratios between variables
5. **Cyclical encoding** — sin/cos of day-of-year, month
6. **Cumulative features** — running sums over season

## Rule 3: No Synthetic Fallbacks

**Law:** If real data is unavailable, raise `DataSourceConnectionError`. Never return fabricated values.

**Why:** Farmers make planting decisions based on HewaSense forecasts. Serving synthetic data that looks real but is fabricated (random normal, hardcoded medians) creates false confidence in worthless predictions.

**Pattern:**
```python
# FORBIDDEN — GOTCHA Law #1 violation
def _baseline_prediction(self, ...):
    return -1.0  # Synthetic marker

# REQUIRED — return None to skip forecast
if self.model is None:
    print(f"ERROR: No model loaded for {trigger_type} forecast.")
    return None  # Skip this forecast entirely
```

## Rule 4: Dynamic Model Resolution

**Law:** Never hardcode model filenames. All model selection is driven by `outputs/models/active_model.json`, which is auto-generated by the training pipeline.

**Why:** Hardcoded model names (like `best_model.pkl`) become stale when models are retrained with different feature counts or architectures. The `best_model.pkl` was trained on 80 features (old schema), causing runtime shape mismatch. Dynamic resolution via `active_model.json` eliminates this class of errors entirely.

**Flow:**
1. Training pipeline (`model_development_pipeline.py` STEP 7) ranks models by test R²
2. Writes best → `primary_model`, second-best → `fallback_model` in `active_model.json`
3. `forecast_service.py` `load_model()` reads `active_model.json` at startup
4. If `active_model.json` is missing → **fail explicitly** (GOTCHA Law #1, no fallbacks)

**Check:** `forecast_service.py` `load_model()` — must read from `active_model.json`, must NOT contain hardcoded model filenames.

## Rule 5: UTC Everywhere

**Law:** All datetime objects must be timezone-aware, using UTC.

**Pattern:**
```python
# FORBIDDEN
from datetime import datetime
now = datetime.now()

# REQUIRED
from datetime import datetime, timezone
now = datetime.now(timezone.utc)
```

## Rule 6: Contract Compliance

**Law:** `PILOT_LOCATION_IDS = [7, 8]` (Ifakara TC = 7, Mlimba DC = 8) everywhere. Payout threshold = 0.75. Canonical sources are the code, not documentation:

- `PILOT_LOCATION_IDS` → `backend/app/services/risk_service.py`
- `PILOT_ZONE_IDS` → `backend/app/services/evaluation_service.py` (imported by `basis_risk_service`)
- Payout threshold → `backend/app/services/pipeline/orchestrator.py` (`_PROB_THRESHOLDS`)

**Why:** Frontend previously hardcoded `locationId: 1` while Backend used `PILOT_LOCATION_ID = 6`. Silent mismatches return wrong data without error. The single-coordinate pilot was replaced by a two-zone split on 2026-04-14 after ground-truth yield data revealed the original Morogoro city coordinates were 120+ km from the actual Kilombero Basin.

**Deprecated:**
- `PILOT_LOCATION_ID = 6` (single Morogoro city) — replaced 2026-04-14
- Any reference to `state.json → shared_contract` — `state.json` was never wired to runtime

## Rule 7: Ingestion TZ Contract (2026-04-16)

**Law:** Ingestion modules use tz-naive `datetime.date` everywhere. See `backend/utils/dates.py` for the full contract.

**Why:** Mixing tz-aware and tz-naive values at comparison sites produced four production bugs (Apr 6/7/9/16 of 2026). Each previous fix was a local patch; the unified contract eliminates the class of bug.

**Pattern:**
```python
# FORBIDDEN in ingestion modules
end_date = datetime.now(timezone.utc)    # tz-aware default
end_date.tz_localize(None)               # ad-hoc reconciliation
end_date.replace(tzinfo=timezone.utc)    # same

# REQUIRED
from utils.dates import as_date, last_complete_month
end_date = as_date(end_date, default=date.today())
cap = last_complete_month()
```

**Note:** Application-layer code (APIs, DB writes) still uses UTC-aware `datetime` — Law #4 applies there. The tz-naive `date` contract is specifically for the ingestion layer's monthly-granularity logic.
