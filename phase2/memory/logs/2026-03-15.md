## Issues Diagnosed & Fixed

### 1. Stale Advisory Lock ‚Äî 3 days of missed pipeline runs (CRITICAL)

**Symptom:** No Slack alerts received for Thu Mar 12, Fri Mar 13, or Sat Mar 14. Pipeline appeared healthy (container up, no crash).

**Root cause:** `release_lock()` called `connection.close()` on a SQLAlchemy `QueuePool` connection. Pool returned the connection to the pool rather than closing the underlying Postgres session ‚Äî so session-level advisory lock `objid=123456` remained attached to PID 314489. Every subsequent 06:00 EAT trigger hit "lock already held" and silently skipped. Lock held for 3 days 7h 57m.

**Evidence:**
```
2026-03-11 03:00:53  Released pipeline execution lock (ID: 123456)  ‚Üê last good run
2026-03-12 03:00:00  Pipeline execution lock already held (ID: 123456)  ‚Üê BLOCKED
2026-03-13 03:00:00  Pipeline execution lock already held (ID: 123456)  ‚Üê BLOCKED
2026-03-14 03:00:00  Pipeline execution lock already held (ID: 123456)  ‚Üê BLOCKED
DB: pid=314489 | state=idle | duration=3 days 07:57:22 | lock objid=123456
```

**Fix (immediate):** Killed PID 314489 directly: `SELECT pg_terminate_backend(314489)` ‚Üí confirmed 0 rows holding lock.

**Fix (permanent):** `orchestrator.py:acquire_lock()` ‚Äî switched to `NullPool` (`poolclass=NullPool`) so every `connection.close()` truly terminates the Postgres backend. `release_lock()` ‚Äî added explicit `pg_advisory_unlock` call before close, plus `engine.dispose()` in `finally` block. `__init__` ‚Äî added `self._lock_engine = None` and `self._lock_connection = None` for safe pre-init state.

**Commits:** `55055b8` (NullPool fix), simplified further in same session (init + warning log)

## Learned Behaviors

- **SQLAlchemy `QueuePool` does NOT close the Postgres session on `connection.close()`** ‚Äî it returns to the pool. Session-level advisory locks survive this. Always use `NullPool` for connections that must truly close (e.g. lock-holders). Diagnostic: check `pg_stat_activity` for idle connections holding the advisory lock objid.
- **`mark_ingestion_complete()` must be called ‚Äî it is not called automatically.** The incremental manager only reads tracking on `calculate_fetch_range()`; writing is the caller's responsibility.
- **`heat_stress` is intentionally excluded** ‚Äî the rainfall model cannot generate a defensible temperature-based heat stress probability. The drought-proxy (√ó0.6 scaling) exists in code but must not be used for insurance triggers.
- **Pipeline math: 3 trigger types √ó 4 horizons = 12 forecasts/run.** Not "4 √ó 3". heat_stress is excluded. Horizons are 3, 4, 5, 6 months.
- **Slack alert numbers on Mar 10 and Mar 11 are verified legitimate** ‚Äî cross-checked against server logs line by line.
- **DB total frozen = incremental tracking not writing.** Symptom: every run shows "No tracking record found for [source]" in logs. Check `source_ingestion_tracking` table.
