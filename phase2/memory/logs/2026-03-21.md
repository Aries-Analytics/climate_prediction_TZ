### v3.6 — CDF plain-language explanation + worked examples

CDF (Cumulative Distribution Function) documented with:

- **Definition:** probability that a normally distributed random variable falls ≤ a given value
- **Plain language:** "given what our model predicts and its measured uncertainty, what is the probability that actual rainfall lands below/above the trigger threshold?"
- **Why CDF is correct (vs binary threshold):** grounds uncertainty in real measured RMSE (33.1mm), not an arbitrary assumption; prevents large payouts when prediction is close to the threshold and actual outcome is uncertain
- **Two worked examples:**
  - 45mm predicted vs 120mm drought threshold → CDF(2.27) ≈ 99% drought probability (clear signal)
  - 115mm predicted vs 120mm drought threshold → CDF(0.15) ≈ 56% drought probability (marginal — model uncertainty dominates)
- **Flood inversion** explained: `1 - CDF(...)` because flood triggers when rainfall EXCEEDS threshold (opposite direction)

## Domain Knowledge Added This Session

**Inference chain — single model run → four horizons:** XGBoost executes once per trigger type. The four horizon outputs come from applying different crop-phase thresholds (derived from the Kilombero rice calendar) to the same prediction. This is a pragmatic simplification, not an architectural flaw. Defensible because ENSO/IOD feature set provides genuine multi-month predictive skill. Decision to add horizon-specific models deferred to post-shadow-run Brier Score comparison.

**CDF rationale for parametric insurance:** Using `norm.cdf((threshold - predicted_mm) / rmse)` rather than a binary threshold check is the correct approach because it grounds uncertainty in the model's own measured RMSE. When the prediction is close to the threshold, the probability reflects that uncertainty rather than defaulting to a hard 0/1. This directly affects reserve sizing accuracy (Stage 1) and helps prevent unfunded payout exposure.

**Where HewaSense sits:** Feature-engineered gradient-boosted ML for agronomic trigger calibration. Not cutting-edge research ML (that would be multi-task spatiotemporal transformers trained on 100TB+ data). Cutting-edge for applied climate risk in Sub-Saharan Africa with 26 years of historical data and a specific parametric insurance application. The design choices are deliberate — interpretability over marginal accuracy gain for a regulated insurance context.

**Three constraints limiting insurance scale:** Every ML improvement reduces one of: (1) basis risk, (2) reserve mispricing, or (3) reinsurance friction. The improvement roadmap is structured around these three constraints, not just model accuracy.
