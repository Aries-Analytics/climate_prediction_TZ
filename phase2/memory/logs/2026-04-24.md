## INCIDENT: Database Destruction During Frontend Deployment

### What Happened
Artemis (remote agent) was asked to fix slow frontend dashboard loading times. During deployment, ran `docker compose -f docker-compose.prod.yml down --remove-orphans` on the production server. This command:
1. Identified old dev containers (`climate_db_dev`, `climate_backend_dev`, `climate_frontend_dev`) as "orphans"
2. Removed them, including `climate_db_dev` which held the `phase2_postgres_data_dev` volume
3. The volume containing **9 days of live shadow run data** (Apr 16-24) was destroyed

### Data Lost (Irreplaceable)
- `forecast_logs` — 9 days of shadow run forecasts (Day 1-9 of 90)
- `pipeline_executions` — 9 days of pipeline run history
- `source_ingestion_tracking` — incremental ingestion state for all 5 sources
- `forecasts` — active forecasts generated during shadow run
- `forecast_validation` — any matched prediction/outcome records

### Data Restored (From Files)
- `climate_data` — 1,872 records reloaded from `/outputs/processed/master_dataset.csv`
- `model_metrics` — 4 models reloaded from `/outputs/models/latest_training_results.json`
- `locations` — 8 locations re-seeded via `scripts/seed_locations.py`
- `users` — 3 users re-seeded via `scripts/seed_users.py`

### Shadow Run Impact
- Shadow run reset to Day 0 of 90
- Projected completion extended by 9 days (gap handling is automatic via `shadow_run.py:projected_end_date()`)
- `SHADOW_RUN_START` remains Apr 16 — code counts actual DB records, not calendar days
- Next pipeline run: 6AM EAT Apr 25 (will be Day 1 again)

## Performance Fixes Applied (Before the Incident)

### Frontend — Parallel API Fetches
- **ForecastDashboard.tsx**: 4 sequential fetches (`fetchForecasts`, `fetchRecommendations`, `fetchPortfolioRisk`, `fetchPilotLocations`) wrapped in `Promise.all()`. Saves ~500ms.
- **RiskManagementDashboard.tsx**: 2 sequential fetches (`fetchPortfolioData`, `fetchRecommendations`) wrapped in `Promise.all()`.
- **ModelPerformanceDashboard.tsx**: 2 sequential fetches (`fetchModels`, `fetchValidationMetrics`) wrapped in `Promise.all()`.
- Removed duplicate `setIsLoading(true/false)` from individual fetch functions (now handled by wrapper).

### Frontend — Buffer Polyfill
- Added `buffer` npm package to fix `Uncaught TypeError: Failed to resolve module specifier "buffer/"` caused by Plotly's `typedarray-pool` dependency.
- Configured in `vite.config.ts` with `resolve.alias` and `define.globalThis.Buffer`.

### Frontend — Axios Cache TTL
- `axiosInstance.ts`: Bumped in-memory GET cache TTL from 2 minutes to 5 minutes.

### NGINX — Compression & Caching
- `nginx.conf`: Added gzip compression (JS, CSS, JSON, SVG), 1-year cache for hashed static assets, 30-day cache for fonts/images, no-cache for index.html, proxy timeouts (15s connect, 30s read/send).

### Docker — Build Memory Fix
- `Dockerfile`: Added `NODE_OPTIONS="--max-old-space-size=4096"` to prevent OOM during Vite build (Plotly.js causes heap exhaustion at default 1.7GB).

## Pipeline Schedule Fix

### Problem
Code had inconsistent defaults: `scheduler.py` defaulted to `0 3 * * *` with `Africa/Dar_es_Salaam` timezone (3AM EAT), while `run_scheduler.py` defaulted to `0 3 * * *` with `UTC` (6AM EAT). The server `.env` had `PIPELINE_SCHEDULE=0 3 * * *` with `PIPELINE_TIMEZONE=Africa/Dar_es_Salaam`.

### Fix
- Server `.env`: Changed to `PIPELINE_SCHEDULE=0 6 * * *`
- `scheduler.py`: Default changed to `0 6 * * *` with `Africa/Dar_es_Salaam`
- `run_scheduler.py`: Default changed to `0 6 * * *` with `Africa/Dar_es_Salaam`
- Scheduler container recreated; confirmed next run: `2026-04-25 06:00:00+03:00`

## Lessons Learned

1. **Never tear down a full Docker stack to fix a single service.** `docker compose up -d <service>` is always sufficient.
2. **Always check what volumes are attached to running containers** before any container removal.
3. **Take a backup before any deployment** — even "simple" frontend changes.
4. **Data that exists as files (models, CSVs) should be reloaded immediately** after DB loss — don't tell the user to "wait for the pipeline."
5. **Don't speculate about pre-existing bugs** when you caused the current problem.
