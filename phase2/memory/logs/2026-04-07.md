### Pipeline Failure ‚Äî Apr 7 Run (06:00 EAT)

Slack alert: era5 failed. chirps, nasa_power, ndvi, ocean_indices all succeeded ‚Äî yesterday's tz-naive fix confirmed clean.

**Root cause identified and fixed:**

`era5_ingestion.py` line 408 used `hasattr(end_date, 'tzinfo')` to detect whether `end_date` needed timezone normalisation before comparing against `era5_safe_end` (a tz-aware `datetime`). However, plain `date` objects (returned by `incremental_manager.calculate_fetch_range()`) do not have a `tzinfo` attribute ‚Äî `hasattr` returned `False`, the normalisation block was skipped entirely, and `date > datetime` raised `TypeError: can't compare datetime.datetime to datetime.date` at line 410.

This is a different bug from Apr 6 (same symptom family ‚Äî type mismatch on date comparison ‚Äî but different code path and different type pair: `date` vs `datetime`, not `tz-naive Timestamp` vs `tz-aware Timestamp`).

**Fix (commit `c77be7a`):**
- Replaced `hasattr(end_date, 'tzinfo')` guard with `isinstance(end_date, datetime)` check
- If plain `date`: promote with `datetime.combine(end_date, time.min).replace(tzinfo=timezone.utc)`
- If `datetime` but tz-naive: existing `.replace(tzinfo=timezone.utc)` path
- Added missing `time` to `from datetime import` line
- Deployed: git push ‚Üí server pull ‚Üí `docker restart climate_pipeline_scheduler_dev`
- Scheduler healthy, next run: `2026-04-08 06:00:00+03:00`

**All four sources (chirps, nasa_power, ndvi, era5) should succeed tomorrow.**
