## Bug Fix: Evidence Pack "Days Completed" Shows 1

### Problem
Evidence Pack dashboard showed "Days Completed: 1" when the shadow run had been running for 2 days (Apr 14 + Apr 15).

### Root Cause
Frontend calculated days as `Math.round(total_forecast_logs / forecasts_per_day)` = `Math.round(24 / 24)` = 1. This integer division approach only works when each day generates exactly `forecasts_per_day` logs AND the total is accumulated — with only 24 logs from today's run, it shows 1.

### Fix
1. **Backend** (`evidence.py`): Added `valid_run_days` (count of distinct calendar dates with forecast logs filtered by `SHADOW_RUN_START`) and `target_days` to `/execution-log` response
2. **Backend** (`evidence.py`): Added `SHADOW_RUN_START` date filter to `total_forecast_logs` query (was unfiltered)
3. **Frontend** (`EvidencePackDashboard.tsx`): Updated `ShadowRunProgress` interface with `valid_run_days` and `target_days`. "Days Completed" now reads directly from API instead of deriving via division.

**Files:** `backend/app/api/evidence.py`, `frontend/src/pages/EvidencePackDashboard.tsx`

## Critical Bug Fix: 12 Forecasts/Day Instead of 24

### Problem
Shadow run generating only 12 forecasts/day (1 location × 3 triggers × 4 horizons) instead of 24 (2 locations × 3 triggers × 4 horizons). Both Apr 14 and Apr 15 runs affected.

### Root Cause — Duplicate Ingestion Module Directories
The codebase has TWO copies of the ingestion modules:
- `phase2/modules/ingestion/` — mounted into scheduler container via `docker-compose.dev.yml` line 81: `./modules:/app/modules:ro`
- `phase2/backend/modules/ingestion/` — lives inside the backend build context

When the two-zone split was implemented (commit `6a4f32b`, Apr 13), only `backend/modules/ingestion/` was updated with `PILOT_LOCATIONS` loop. The top-level `phase2/modules/ingestion/` — the one actually deployed — kept hardcoded Morogoro coordinates:
```
morogoro_lat = -6.8211
morogoro_lon = 37.6595
```

This meant:
1. All 5 ingestion sources stored climate data at Morogoro coordinates (120+ km from actual pilot zones)
2. `forecast_service.py:prepare_features()` searched for climate data within ±0.01° of Ifakara TC (-8.1333°S, 36.6833°E) and Mlimba DC (-8.0167°S, 35.9500°E) — found nothing
3. Only pre-existing locations with legacy data could generate forecasts → 12/day instead of 24/day

### Fix
Updated all 5 files in `phase2/modules/ingestion/`:
- `chirps_ingestion.py`
- `era5_ingestion.py`
- `nasa_power_ingestion.py`
- `ndvi_ingestion.py`
- `ocean_indices_ingestion.py`

Each now has the `PILOT_LOCATIONS` loop storing data at both zone coordinates. The deployed (modules/) versions also retain the Apr 6 tz-naive fix that the backend versions are missing.

### Architectural Debt
Having two copies of the same ingestion code is a maintenance hazard — this is exactly how this bug was introduced. Needs consolidation: either change docker-compose to mount `./backend/modules` or eliminate the duplicate. Added to Q2 roadmap.

### Deployment Required
After commit+push: server needs `git pull` + `docker restart climate_pipeline_scheduler_dev`. First corrected run will backfill climate data for both zones, then generate 24 forecasts/day going forward. Existing 24 forecast logs (from 2 days) may need to be kept or wiped depending on whether the single-location forecasts are considered valid.
