- `Ocean Indices: Invalid comparison between dtype=datetime64[ns] and Timestamp`

After backfill, DB has 24 forecast logs (12 per zone). Correct count achieved, but 2/5 ingestion sources silently broken when called with `end_date=None`.

## Permanent Fix: TZ Contract for Ingestion Layer

### Root Cause (not symptoms)
All 5 ingestion modules mixed tz-aware defaults (`datetime.now(timezone.utc)`) with tz-naive comparison targets. This required inline reconciliation patches (`tz_localize`, `replace(tzinfo=...)`, isinstance dances) at every comparison site. Each previous fix addressed ONE path and left others broken — classic whack-a-mole.

### The Contract (documented in `backend/utils/dates.py`)
- **All date parameters:** tz-naive `datetime.date`
- **Defaults:** `date.today()`, `date(2010, 1, 1)` — never `datetime.now(timezone.utc)`
- **Internal math:** pure `date` arithmetic (month/year calculations are calendar concepts, not instants — no UTC needed)
- **Pandas bounds:** `pd.Timestamp` (tz-naive), converted only at the DataFrame filtering site
- **Zero tz reconciliation** below the public API boundary

### Helpers (`backend/utils/dates.py`)
- `as_date(x, default)` — coerces any input (`date`, `datetime`, `Timestamp`, `str` ISO, `None`) to tz-naive `date`. Raises `TypeError` on bad input (Law #1: explicit failure)
- `last_complete_month()` — returns last day of previous calendar month as tz-naive `date`
- `subtract_months(d, months)` — calendar-month arithmetic for ERA5 lag

### Changes Applied
- **NASA POWER:** Removed tz-aware default + partial-month tz patch. 7 lines of tz dance → 1 line `as_date()`
- **Ocean Indices:** Same treatment
- **CHIRPS:** Removed 2 inline tz_localize patches + tz-aware default
- **NDVI:** Same treatment
- **ERA5:** Replaced UTC-anchored ERA5T lag math with pure `date` arithmetic via `subtract_months()`. Removed 6 lines of tz reconciliation. Both `fetch_era5_data()` and `ingest_era5()` cleaned.
- All 5 modules: type hints updated `datetime→date`; module docstrings reference the contract.
- **Pandas boundary:** All 5 modules now convert to `pd.Timestamp(date)` exactly at the DataFrame filtering site, preserving the contract above the boundary.

## Runbook: Post-Reset Backfill (Documented for Future)

When doing a DB wipe that changes pilot zone coordinates, the runbook must include a manual backfill step BEFORE the first scheduled run:

```bash
# On server
docker exec climate_pipeline_scheduler_dev python -c "
from app.core.database import SessionLocal
from datetime import date
from modules.ingestion.chirps_ingestion import ingest_chirps
from modules.ingestion.nasa_power_ingestion import ingest_nasa_power
from modules.ingestion.ndvi_ingestion import ingest_ndvi
from modules.ingestion.era5_ingestion import ingest_era5
from modules.ingestion.ocean_indices_ingestion import ingest_ocean_indices

db = SessionLocal()
for name, fn in [('chirps', ingest_chirps), ('nasa', ingest_nasa_power),
                 ('ndvi', ingest_ndvi), ('era5', ingest_era5), ('ocean', ingest_ocean_indices)]:
    try:
        f, s = fn(db, start_date=date(2010, 1, 1), end_date=None, incremental=False)
        print(f'{name}: fetched={f}, stored={s}')
    except Exception as e:
        print(f'{name}: FAILED: {e}')
db.close()
"
```

Then verify climate_data count at pilot zone coordinates before the first scheduled run.
