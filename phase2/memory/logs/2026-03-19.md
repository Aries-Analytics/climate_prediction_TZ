## Learned Behaviors Added This Session (Session 1)

**Pipeline Completion Timestamp Trap:** `ForecastLog.issued_at` is the write time at pipeline *completion*, not the scheduled start time. A 41-minute run starting at 03:00 UTC writes entries at 03:41 UTC — this is correct. Never assess legitimacy from `issued_at` alone; cross-check `pipeline_executions.started_at`.

**Prod-vs-Dev Compose Drift Trap:** Active stack is always `docker-compose.dev.yml`. If prod containers appear (`climate_db_prod`, `climate_frontend_prod`, `phase2-backend-N`), that is wrong state — stop them and restore dev compose. The startup script at `/opt/hewasense/start.sh` must always reference dev compose.

**Destructive Action Gate (ALL sensitive work):** Never execute a destructive or irreversible action (delete DB records, drop containers, wipe data, force-push, remove files) on user instruction alone. Required sequence: (1) verify the claim independently, (2) show the user what will be affected and the evidence, (3) if data is correct — push back explicitly and state why. Do not comply silently to avoid friction. This applies to shadow run DB, server infrastructure, git history, model artifacts, configs — anything hard or impossible to reverse.

### Payout Mechanism Clarified — Option A Confirmed (Binary Trigger)

**Issue identified:** Business case docs (internal + external), `PAYOUT_CALCULATION_MODEL.md`, and `PARAMETRIC_INSURANCE_FINAL.md` all used `PILOT_FARMERS × probability × rate` as if it were the actual farmer payout. This is incorrect — it is the **reserve sizing formula** (probability-weighted financial exposure while forecast is outstanding). The actual payout is binary.

**User confirmed: Option A — zone-level binary trigger.**

**Two-stage mechanism (now documented across all parametric product docs):**

| Stage | Formula | Purpose |
|---|---|---|
| Reserve sizing (forecast outstanding) | `PILOT_FARMERS × probability × rate` | Capital earmarked against future liability |
| Confirmed payout (threshold observed) | `PILOT_FARMERS × rate` if fires, else $0 | Actual payment — ALL enrolled farmers in zone |

**Design rationale:** Zone index is either breached or it isn't. All enrolled farmers in the zone bear the same risk and receive the same fixed payout. No individual assessment, no partial payments. The probability is a reserve-management number, not a payout selector.

## Domain Knowledge Added This Session

**Payout Mechanism — Option A / Binary Trigger:** HewaSense payout is zone-level, binary. `PILOT_FARMERS × probability × rate` is a reserve-sizing formula (Stage 1), not a farmer payment. When observed data confirms the threshold was breached, ALL enrolled farmers in the zone receive the fixed rate (Stage 2). The 82% probability determines how much to hold in reserve; the confirmed observation determines whether the trigger fires at all. Paying 82% of $60 per farmer would require identifying which farmers were affected — eliminating exactly the advantage of parametric insurance.
