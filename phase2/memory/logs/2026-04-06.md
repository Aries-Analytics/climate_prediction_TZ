## Session Notes

### Pipeline Failure — Apr 6 Run (06:00 EAT)

Slack alert: chirps, nasa_power, era5, ndvi all failed. ocean_indices succeeded.

**Root cause identified and fixed:**

The partial-month cap (added Apr 1, commit `11faebd`) compares `end_date` (tz-naive pandas Timestamp, from `incremental_manager.calculate_fetch_range()` → `date.today()`) against `last_complete_month_end` (tz-aware pandas Timestamp, built from `datetime.now(timezone.utc)`). The comparison only fires when `end_date > last_complete_month_end` — i.e. when the incremental manager returns a current-month date. Apr 6 was the first day this path was reached, triggering `TypeError: Cannot compare tz-naive and tz-aware timestamps` in chirps, nasa_power, and ndvi.

ERA5 was not affected — it already normalises `end_date` to UTC-aware before comparison via `end_date.replace(tzinfo=timezone.utc)`.

**Fix (commit `40b1bc7`):**
- Replaced `now_utc = datetime.now(timezone.utc)` + tz-aware `pd.to_datetime` with pure date arithmetic in all 3 modules:
  ```python
  _today = date.today()
  last_complete_month_end = pd.to_datetime(date(_today.year, _today.month, 1) - timedelta(days=1))
  ```
- Added missing `date` import to `nasa_power_ingestion.py` and `ndvi_ingestion.py`
- Verified fix live inside container via `docker exec` grep
- Deployed: git push → `root@37.27.200.227` pull → `docker restart climate_pipeline_scheduler_dev`
- Scheduler confirmed healthy, next run: `2026-04-07 06:00:00+03:00`

**Why "0 new records" tomorrow is correct:**
All four monthly sources are capped to March 31. If March is already fully ingested, the incremental manager returns start = end → 0 records fetched. This is expected, not a failure.
